# Промт: анализ проекта для резюме

Промт для генерации файла-разбора проекта в `projects/<slug>.md` — в том же формате,
что `projects/quick.md` и `projects/code-kata.md`. Результат — источник честных фактов
и готовых буллетов для `cv/data.yaml`.

## Как использовать

1. Подставь переменные в блоке `<input>` ниже (путь к репозиторию и целевую роль).
2. Передай весь промт ассистенту с доступом к коду этого репозитория (Claude Code / агент).
3. Сохрани ответ в `projects/<slug>.md`.

---

## Промт (копировать целиком)

````text
Ты — карьерный инженер-аналитик. Проанализируй репозиторий проекта с точки зрения
карьерного позиционирования под конкретную роль и составь файл-разбор на русском языке.

<input>
ПРОЕКТ:        {{путь к локальному репозиторию или URL}}
ЦЕЛЕВАЯ РОЛЬ:  {{например: Junior AI Backend / ML Engineer}}
ССЫЛКА НА РЕПО: {{github.com/... или "приватный"}}
</input>

ПРАВИЛА РАБОТЫ (обязательны):
1. Сначала прочитай реальный код: README, структуру каталогов, точки входа, конфиги
   (pyproject/Taskfile/Dockerfile/CI), ключевые модули. Опирайся ТОЛЬКО на то, что
   реально есть в репозитории.
2. Жёсткая честность. В начале дай «ключевой честный вывод»: чем проект ЯВЛЯЕТСЯ
   и чем НЕ является. Если проект не соответствует громкому ярлыку (напр. это не
   «ML-проект», а «scientific computing» или «учебный репозиторий») — скажи прямо
   и предложи корректное позиционирование. Всё, что вскроется на первом техническом
   вопросе, должно быть названо до того.
3. Не приписывай несуществующее. В каждом разделе явно перечисляй, чего в проекте
   НЕТ (тесты, CI/CD, БД, API, async, деплой, LLM/RAG — что применимо), и помечай,
   что это закрывается другими проектами, а не выдумывается.
4. Любые числа/метрики, которых нет в коде или истории, помечай `[уточнить]`
   (например: ускорение, размер образа, место на лидерборде, прирост от тюнинга).
   Не выдумывай цифры.
5. Ссылайся на конкретные пути в коде (`api/routes.py`, `src/.../cache.py`) —
   это доказывает, что разбор основан на реальном чтении.
6. Буллеты для резюме пиши по правилам из `docs/resume-rules.md`:
   формула X-Y-Z (что сделал → измеримый результат → как), сильный глагол в начале,
   метрика в каждом ключевом буллете (или `[уточнить]`), без клише и «воды».
7. Оценивай всё через призму ЦЕЛЕВОЙ РОЛИ: что ценно, что смежно-ценно, что нерелевантно.

ФОРМАТ ВЫВОДА (Markdown, ровно эти секции и порядок):

# Анализ проекта <ИМЯ> для роли <ЦЕЛЕВАЯ РОЛЬ>

> Дата: <YYYY-MM-DD>. Пометки `[уточнить]` — данные, требующие подтверждения.

**Ключевой честный вывод:** <1 абзац: чем проект является и чем нет; корректное
позиционирование под роль; самый серьёзный риск/пробел.>

---

## 1. Ценность для роли
Сгруппируй по направлениям, релевантным роли (напр. Backend / AI-ML / Инженерная
культура). Для каждого факта — ссылка на код. В конце — чего работодатель здесь
НЕ увидит.

## 2. Backend-сильные стороны (честно)
Таблица `| Область | Что есть в коде |` (или `| Есть | Нет (в этом репо) |`).
Отдельно — блок «Чего нет (не приписывать)» с явным списком пробелов и пометкой
самого критичного для собеседования.

## 3. AI/ML-сильные стороны — честно
Что честно можно заявить как ML / ML-смежное. Если прямого ML нет — так и напиши,
и покажи смежные сигналы (численные методы, вероятностное моделирование, vector
search и т.п.). Чего нет — перечислить.

## 4. Инженерное мышление — что рассказывать
Trade-offs и архитектурные решения, которые проект демонстрирует (точность/скорость,
безопасность/удобство, расширяемость, ownership). По пунктам, с сутью решения.

## 5. Bullet points для резюме
Нумерованный список готовых буллетов по правилам `docs/resume-rules.md`.
Невыводимые числа — с пометкой `[уточнить]`.

## 6. Навыки для резюме
Сгруппированные по категориям (Python / Backend / AI-ML / Данные / БД / DevOps /
Тестирование / ...). Явно отметь навыки, которые этим репозиторием НЕ подтверждаются
(их закрывают другие проекты), и те, что заявлять пока рано (напр. тестов нет).

## 7. ATS-ключевые слова
Список ключевиков, ПОДТВЕРЖДАЕМЫХ этим проектом. Отдельной строкой — ключевики
под целевую роль, которые надо подтверждать ДРУГИМИ проектами.

ОПЦИОНАЛЬНО (добавляй, если уместно для проекта):

## 8. GitHub README — что улучшить
## 9. Рассказ на собеседовании
(проблема → что сделал → технологии → сложности → решения → что улучшил бы)
## 10. Открытые вопросы (что уточнить)
## Главная рекомендация
1–2 предложения: как связать проект с остальными в портфолио и какое усиление
(на 1–2 недели) даст максимальный эффект.

ТОН: инженерный, конкретный, без маркетинговой воды. Язык — русский.
````

## Заметки

- Slug файла = короткое имя проекта в kebab-case (`projects/ai-notes-api.md`).
- После генерации переноси в `cv/data.yaml` только проверенные факты; всё с
  `[уточнить]` — либо подтверди числами, либо не включай (правило №2 из
  `docs/resume-rules.md`).
- Промт намеренно требует читать реальный код: разбор без ссылок на пути в репозитории
  считается невыполненным.
