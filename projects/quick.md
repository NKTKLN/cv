# Анализ проекта QUICK для роли Junior AI Backend / ML Engineer

**Ключевой честный вывод:** QUICK — это не ML-проект в классическом смысле (нет обученных моделей, LLM, embeddings, RAG). Это **проект научных вычислений (scientific computing)**: моделирование систем массового обслуживания через аналитические методы, численное интегрирование ОДУ и Монте-Карло симуляцию. Для роли ML Engineer это **сильный смежный сигнал** — линейная алгебра, NumPy/SciPy, стохастическое моделирование, численная устойчивость — но позиционировать его как "ML-проект" нельзя, это вскроется на первом же техническом вопросе. Правильное позиционирование: **"вычислительный backend-сервис с математическим ядром"**. А вот backend-составляющая у проекта — действительно сильная и честная.

---

## 1. Ценность для роли

**Backend-навыки (сильно представлены):**

- REST API на FastAPI: `api/main.py`, `api/routes.py` — lifespan-инициализация ресурсов, версионирование (`/api/v1`), health-check эндпоинт.
- Pydantic-схемы запросов/ответов (`api/schemas.py`) с StrEnum, валидация на двух уровнях (схема + доменная `params.validate()`).
- Обработка ошибок с корректными HTTP-кодами: 422 для невалидных параметров, 500 для внутренних ошибок, с логированием через loguru (`api/routes.py:simulate`).
- Кэширование вычислений в DuckDB через собственный декоратор `duckdb_cache` (`src/quick/db/cache.py`) — ключ из аргументов метода и вложенных атрибутов объекта, graceful degradation при сбое кэша (ошибка кэша не роняет запрос).
- **Безопасность:** собственный `SafeUnpickler` с белым списком классов (`src/quick/utils/safe_unpickler.py`) — защита от RCE при десериализации pickle. Это редкость для junior-проектов и отличная тема для собеседования. Плюс gitleaks в pre-commit.
- Multi-stage Dockerfile с uv, non-root пользователем (фиксированный UID для Kubernetes), HEALTHCHECK, кэшированием слоёв зависимостей.
- Конфигурация через pydantic-settings и `.env`.

**Математические / "ML-смежные" навыки:**

- Три метода решения одной задачи: аналитический (матричная экспонента), численный (система ОДУ Колмогорова через `scipy.solve_ivp`, RK45), имитационный (Монте-Карло).
- Гибридный солвер `AnalyticalMergedProbabilitySolver`: NumPy для скорости + mpmath для точности, с автоматическим детектом численно некорректных участков и пересчётом их с повышенной точностью. Это осознанная работа с trade-off "точность vs производительность" — то, что реально ценится в ML-инженерии.
- MAP-потоки (Markovian Arrival Process), времязависимые интенсивности (LOCF), генерация стохастических матриц.

**Что важно работодателю:** чистая слоистая архитектура (api / web / domain / services / db / utils), strict mypy + ruff с большим набором правил, conventional commits + commitizen (144 коммита, semver 2.1.0), pip-audit и deptry — это сигнал зрелой инженерной культуры, нетипичной для junior.

## 2. Backend-сильные стороны (резюме по фактам)

| Область | Что есть в коде |
|---|---|
| API | FastAPI, версионированные роуты, `/meta` справочник enum'ов, OpenAPI из коробки |
| Валидация | Pydantic v2 схемы + доменная валидация, маппинг ошибок в 422 |
| БД | DuckDB как кэш-хранилище, SQL-схема, singleton-клиент |
| Архитектура | Domain-driven слои, фабрики (`system_factory`, фабрики солверов), паттерн Strategy для солверов |
| Docker | Multi-stage, uv, non-root, healthcheck, .dockerignore |
| Качество | mypy strict (`disallow_untyped_defs`), ruff (security-правила `S`, complexity `C90`), pre-commit, gitleaks, pip-audit, deptry |
| Логирование | Структурированное через loguru, конфигурируемые уровни и пути |

**Чего нет (не приписывать):** авторизации, асинхронных задач/очередей (эндпоинт `/simulate` синхронный), CI/CD (нет `.github/workflows` — есть только pre-commit локально), **тестов** (директории `tests/` нет, хотя конфиг ruff её упоминает). Отсутствие тестов — самый серьёзный пробел для собеседования, его стоит закрыть до рассылки резюме.

## 3. AI/ML-сильные стороны — честно

Прямых ML-компонентов нет. Что **честно можно** заявить как ML-смежное:

- Стохастическое моделирование (Монте-Карло), марковские процессы — фундамент для понимания probabilistic ML.
- Численные методы: интегрирование ОДУ, работа с матрицами переходов, контроль численной устойчивости и точности (mpmath arbitrary precision).
- NumPy/SciPy на продвинутом уровне, конвертация numpy-типов в JSON-сериализуемые (`api/converters.py`) — типовая задача model serving.
- Архитектура inference-подобного сервиса: запрос с параметрами → валидация → вычислительный движок → кэширование результатов → структурированный ответ. Это та же форма, что у ML inference API, и так это можно подать: *"спроектировал API по паттерну inference-сервиса"*.

Если хочется усилить ML-профиль, проще всего добавить в проект небольшой ML-компонент (например, суррогатную модель, предсказывающую характеристики СМО по параметрам, обученную на данных собственного симулятора) — это органично и даст полный цикл данные → обучение → serving.

## 4. Инженерное мышление — что рассказывать

- **Trade-off точность/скорость:** mpmath точен, но медленный; NumPy быстрый, но накапливает ошибку на жёстких системах. Решение — гибридный солвер, который считает быстро и пересчитывает с высокой точностью только проблемные участки.
- **Trade-off безопасность/удобство:** pickle нужен для кэширования numpy/mpmath объектов, но pickle небезопасен → написан SafeUnpickler с whitelist.
- **Производительность:** дорогие вычисления кэшируются в DuckDB по семантическому ключу; кэш отключаем через конфиг; сбой кэша не ломает основной поток.
- **Расширяемость:** фабрики и базовые классы — новый тип системы или метод расчёта добавляется без правки существующих солверов.
- **Ownership:** 144 коммита, единственный автор, conventional commits, ведение версий — полный цикл от математической постановки до Docker-образа.

1. Разработал backend-сервис для математического моделирования систем массового обслуживания (~7,5 тыс. строк Python), пройдя полный цикл от доменной модели до Docker-развёртывания.
2. Спроектировал REST API на FastAPI с Pydantic-валидацией, версионированием эндпоинтов, обработкой ошибок (422/500) и автогенерируемой OpenAPI-документацией.

4. Создал гибридный решатель, сочетающий NumPy и mpmath: автоматическое обнаружение численно нестабильных участков и их пересчёт с произвольной точностью, что устранило накопление ошибки без потери производительности на стабильных участках [уточнить ускорение в цифрах].
5. Разработал декоратор кэширования результатов тяжёлых вычислений в DuckDB с формированием ключа из параметров вызова и graceful degradation при сбоях кэша.
6. Устранил риск выполнения произвольного кода при десериализации кэша, реализовав SafeUnpickler с белым списком допустимых классов.
7. Реализовал Монте-Карло симуляцию марковских систем (MAP-потоки) с времязависимыми интенсивностями на NumPy.
8. Настроил multi-stage Docker-сборку (uv, non-root user, healthcheck), сократив размер итогового образа и подготовив контейнер к запуску в Kubernetes [уточнить размер образа].
9. Внедрил инженерные практики качества: strict mypy, ruff (включая security-правила), pre-commit с gitleaks и pip-audit, conventional commits с автоматическим версионированием.
10. Спроектировал слоистую архитектуру (domain / services / db / api / web), изолировав математическое ядро от веб-интерфейса (Streamlit) и REST API, что позволило двум клиентам использовать одну кодовую базу.
